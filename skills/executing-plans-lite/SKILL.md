# Executing Plans Lite

A lightweight workflow for executing tasks defined in a single markdown PRD
file (e.g., `.tasks/prd-[feature].md`), keeping the file updated, and
archiving completed tasks immediately to maintain context efficiency.

**Announce at start:**
"I'm using the executing-plans-lite skill to implement this plan."

## When to Use

- Use when:
  - The user explicitly requests the agent to implement a task or checklist
    point in a PRD under `.tasks/`.
  - Prerequisite: A lightweight PRD/plan exists in `.tasks/` (created via
    `writing-plans-lite`).

## The Process

### Step 1: Load and Review PRD

1. Read the active PRD file in `.tasks/`.
2. Inspect the **Active Dashboard** (Section 2) and **Active Task Details**
   (Section 3).
3. Review the active task critically. If you identify any questions, gaps, or
   concerns, raise them with the user *before* starting.
4. If there are no active tasks, ask the user for what to do next.

### Step 2: Execute and Progress

1. Mark the current task as In-Progress (`[/]`) in the Active Dashboard.
2. Follow the checklist items exactly. For each item, implement the code
   changes and run tests/verification.
3. Check off completed items in the task's checklist in Section 3. Do not skip
   verifications.

### Step 3: Archive Completed Tasks (Critical)

1. Once a task is fully completed, mark it as completed (`[x]`) in Section 2.
2. Cut the task entry from Section 2 (Active Dashboard) and its
   details/checklist from Section 3 (Active Task Details).
3. Append both to **Section 5 (History / Archive)**.
   > [!IMPORTANT]
   > Archiving completed tasks immediately keeps the active PRD short,
   > preventing token bloat and maintaining model focus.

### Step 4: Pause and Safe-Check

- If you need to pause to gather context or wait for user input, you can
  optionally append a final safeguard item to the task checklist:
  `- [ ] Wait for further instruction from user`.

## When to Stop and Ask for Help

**STOP executing immediately and ask the user (do not guess) when:**

- You hit a blocker (missing dependency, failing test, or unclear instruction).
- The PRD has critical gaps preventing you from starting or continuing.
- Verification fails repeatedly.

## Remember

- Update the PRD file status at the end of each turn.
- Archive completed tasks immediately.
- Run verifications as specified, and don't force through blockers.
